# Auto-collapsing generated / vendored files

A PR full of `bun.lock` churn buries the changes worth reading. GitHub solves this
by **collapsing generated and vendored files by default** in the diff — you see a
one-line "this is generated" stub and click to expand. The review pane does the
same. This note records *which* files collapse and *why*; the rules live in
[`src/server/generated.ts`](../src/server/generated.ts) (`classifyFile`), tested in
[`tests/generated.test.ts`](../tests/generated.test.ts).

## How GitHub decides

GitHub runs every file through [github-linguist](https://github.com/github-linguist/linguist):

- **`lib/linguist/generated.rb`** — name/path rules (lockfiles, `*.min.js`,
  `*.js.map`, protobuf/`.pb.go`, `.designer.cs`, …) plus *content* sniffing (first-line
  `// Code generated …` markers, a minified file's average line length > ~110 chars,
  source-map magic comments, and many tool-specific signatures).
- **`lib/linguist/vendor.yml`** — a big regex list of third-party paths:
  `node_modules/`, `bower_components/`, `vendor/`, `dist/`, `Godeps/`, minified
  library files (`jquery*.js`, `bootstrap*.css`), and so on.
- **`.gitattributes`** — a repo overrides the guesses explicitly with
  `path linguist-generated` / `linguist-vendored` (and can opt a file back *out* with
  `=false` / `-linguist-generated`). This is authoritative.

A file flagged by any of these is collapsed in the diff (and dropped from the repo's
language stats — not our concern here).

## What we implement

The review pane only holds a PR's **per-file patch** (filename, ±counts, the unified
hunks) — not whole-file bytes — so linguist's content sniffing mostly doesn't apply.
`classifyFile` covers the parts that *are* decidable from that payload, in priority
order:

1. **`.gitattributes`** (when the server can fetch one at the PR head) —
   `linguist-generated` / `linguist-vendored` win over every built-in rule, and an
   explicit `=false` vetoes them (a repo can un-mark its own lockfile).
2. **Lockfiles** — basename match, case-insensitive: `bun.lock`/`bun.lockb`,
   `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `npm-shrinkwrap.json`,
   `Cargo.lock`, `Gemfile.lock`, `composer.lock`, `poetry.lock`, `Pipfile.lock`,
   `uv.lock`, `pdm.lock`, `pixi.lock`, `deno.lock`, `go.sum`, `Podfile.lock`,
   `Package.resolved`, `flake.lock`, `mix.lock`, `pubspec.lock`, `gradle.lockfile`,
   `Gopkg.lock`, `glide.lock`, `.terraform.lock.hcl`, `MODULE.bazel.lock`,
   `packages.lock.json`.
3. **Vendored trees** — any path segment is `node_modules`, `bower_components`,
   `jspm_packages`, `vendor`, `vendors`, `third_party`, `Godeps`.
4. **Source maps** — basename ends `.map`.
5. **Minified** — basename matches `*.min.{js,css,mjs,cjs}`.
6. **Generated code by name** — `.pb.go`/`.pb.cc`/`.pb.h`, `_pb2.py`/`_pb2_grpc.py`,
   `.designer.cs`/`.designer.vb`, `.g.dart`/`.freezed.dart`.
7. **Minified by line length** — the patch's added lines average > 110 chars (catches
   built bundles like `public/assets/main.js` that aren't named `.min`). This is
   linguist's averaging rule applied to the added side of the patch.
8. **Large diff** — more than 2000 changed lines (GitHub's "large diffs are not
   rendered by default"). The threshold is deliberately high so ordinary files stay
   open.

Each match carries a **reason** (`lockfile`, `vendored`, `minified`, `sourcemap`,
`generated`, `gitattributes`, `large`) that the file header shows, so a collapsed
file always says why — and the operator can expand any of them.

### Deliberately out of scope

Content-marker sniffing that needs the whole file (`// Code generated by …`, jOOQ /
Thrift / PEG.js banners, CoffeeScript shape, etc.) — we'd have to pull every blob to
check. The name/path rules plus the line-length heuristic catch the cases that
actually clutter a PR here (`bun.lock`, built bundles), and `.gitattributes` is the
escape hatch for anything else a repo wants folded.
